# Plan 2.1: Web Push API Implementation

Implement Web Push API — browser notification service in backend for real-time alerts on motion detection and unknown face detection.

## Overview

Add server-side Web Push API support using the `web-push` npm package to send browser push notifications to users when security events occur. This involves VAPID key generation, subscription management, and integration with existing motion and face detection workflows.

## Tasks

### 1. Install and Configure Dependencies
- Install `web-push` package in server/package.json
- Generate VAPID key pair for application identity
- Store keys in environment variables (VAPID_PUBLIC_KEY, VAPID_PRIVATE_KEY)
- Add VAPID subject (mailto: or https://) to config

### 2. Create Notification Service Module
- Create `server/src/services/notificationService.ts`
- Implement `generateVAPIDKeys()` for initial setup
- Implement `sendPushNotification(subscription, payload)` function
- Implement `sendNotificationToUser(userId, payload)` function
- Add retry logic (3 attempts with exponential backoff)
- Add error handling for invalid subscriptions
- Implement subscription cleanup (remove expired/invalid)

### 3. Database Schema for Subscriptions
- Create migration `009_notifications.sql`
- Create `notification_subscriptions` table:
  - id (UUID primary key)
  - user_id (UUID foreign key → users)
  - endpoint (text, unique)
  - keys_p256h (text)
  - keys_auth (text)
  - created_at (timestamp)
  - last_used (timestamp)
  - is_active (boolean)
- Create `notification_logs` table:
  - id (UUID primary key)
  - user_id (UUID)
  - event_id (UUID → events)
  - type (text: motion, face, object)
  - payload (jsonb)
  - sent_at (timestamp)
  - status (text: success, failed, pending)
  - error_message (text, nullable)

### 4. Create TypeORM Models
- Create `server/src/models/NotificationSubscription.ts`
- Create `server/src/models/NotificationLog.ts`
- Define relationships with User and Event models

### 5. API Endpoints for Subscription Management
- Create `server/src/routes/notificationRoutes.ts`
- `POST /api/notifications/subscribe` - Save push subscription
  - Body: { endpoint, keys: { p256h, auth } }
  - Validates subscription format
  - Stores in database
  - Returns subscription ID
- `DELETE /api/notifications/unsubscribe` - Remove subscription
  - Body: { endpoint }
  - Deletes from database
- `GET /api/notifications/subscription` - Get active subscription
  - Returns current user's subscription
  - Used by frontend to check if already subscribed

### 6. Notification Payload Construction
- Create `buildNotificationPayload(event)` function
- Include fields:
  - title: "Motion Detected" or "Unknown Face Detected"
  - body: Camera name, event type, timestamp
  - icon: URL to app icon
  - image: Thumbnail URL from event
  - badge: Small icon URL
  - tag: Unique ID for notification coalescing
  - data: { eventId, cameraId, timestamp }
- Payload size < 4KB (Web Push limit)

### 7. Integrate with Motion Detection
- Modify `server/src/detection/optimizedMotionDetection.ts`
- After event creation, trigger notification:
  ```typescript
  if (eventCreated) {
    await notificationService.notifyMotionEvent(event);
  }
  ```
- Include event thumbnail URL in notification

### 8. Integrate with Face Detection
- Modify `server/src/detection/motionTriggeredDetection.ts`
- After face detection, check if unknown:
  ```typescript
  if (facesDetected === 0 && personsDetected > 0) {
    await notificationService.notifyUnknownFace(event);
  }
  ```
- Only notify for unknown faces (NOTF-02)

### 9. Add Authentication Middleware
- Protect notification routes with JWT auth
- Add user ID from JWT token to subscription
- Ensure users can only manage their own subscriptions

### 10. Testing and Error Handling
- Test subscription save/retrieve
- Test notification send with valid subscription
- Test handling of expired subscriptions (410 Gone)
- Test handling of invalid endpoints
- Add comprehensive logging
- Log all notification attempts to `notification_logs` table

## Technical Notes

- VAPID keys must be persistent across restarts
- Subscriptions are device and browser-specific
- One user can have multiple subscriptions (phone, laptop, etc.)
- Web Push limits payload to 4KB
- Notifications include thumbnail URL, not full image
- Use `tag` field to coalesce multiple notifications of same type
- Handle 410 Gone status by removing invalid subscription
- Use exponential backoff: 1s, 2s, 4s retry delays
