# Plan 2.2: Notification Preferences UI

Build notification preferences UI — per-type toggle and quiet hours configuration.

## Overview

Create a user interface for managing notification preferences, allowing users to:
- Enable/disable notifications by event type (motion, face, object)
- Set quiet hours to suppress notifications during specific time periods
- Configure notification behavior without disabling entirely

This requires database schema, API endpoints, and React UI components.

## Tasks

### 1. Database Schema for Preferences
- Create migration `010_notification_preferences.sql`
- Create `notification_preferences` table:
  - id (UUID primary key)
  - user_id (UUID, unique, foreign key → users)
  - motion_enabled (boolean, default true)
  - face_enabled (boolean, default true)
  - object_enabled (boolean, default true)
  - quiet_hours_enabled (boolean, default false)
  - quiet_hours_start (time, default "22:00")
  - quiet_hours_end (time, default "06:00")
  - quiet_hours_timezone (text, default "Asia/Kolkata")
  - created_at (timestamp)
  - updated_at (timestamp)
- Add unique constraint on user_id
- Create index on user_id for fast lookups

### 2. TypeORM Model
- Create `server/src/models/NotificationPreferences.ts`
- Define entity with @Entity decorator
- Add @Column decorators for all fields
- Add @OneToOne relationship with User
- Add validation rules:
  - Time format validation (HH:MM)
  - Timezone validation (IANA timezone database)
  - Boolean defaults

### 3. API Endpoints
- Add to `server/src/routes/notificationRoutes.ts`:
  - `GET /api/notifications/preferences` - Get user preferences
    - Returns: { motion_enabled, face_enabled, object_enabled, quiet_hours_* }
    - Creates default preferences if not exist
  - `PUT /api/notifications/preferences` - Update preferences
    - Body: { motion_enabled?, face_enabled?, object_enabled?, quiet_hours_*? }
    - Validates time ranges (start < end or overnight)
    - Validates timezone
    - Updates updated_at timestamp
  - `POST /api/notifications/preferences/reset` - Reset to defaults
    - Restores default values
- Protect all routes with JWT authentication

### 4. Frontend API Service
- Add methods to `frontend/src/services/ApiService.ts`:
  - `getNotificationPreferences()` - Fetch preferences
  - `updateNotificationPreferences(data)` - Update preferences
  - `resetNotificationPreferences()` - Reset to defaults
- Handle errors gracefully
- Cache preferences in React state

### 8. Backend Preference Checking
- Modify `server/src/services/notificationService.ts`
- Add `checkNotificationPreferences(userId, eventType)` function
- Query preferences from database
- Check if event type is enabled
- Check if current time is within quiet hours
- Return boolean: shouldNotify
- Use this check before sending notifications
- Implementation:
  ```typescript
  async function shouldNotify(userId: string, eventType: string): Promise<boolean> {
    const prefs = await getPreferences(userId);

    // Check event type enabled
    if (eventType === 'motion' && !prefs.motion_enabled) return false;
    if (eventType === 'face' && !prefs.face_enabled) return false;
    if (eventType === 'object' && !prefs.object_enabled) return false;

    // Check quiet hours
    if (prefs.quiet_hours_enabled) {
      const now = new Date();
      const currentTime = formatInTimeZone(now, prefs.quiet_hours_timezone, 'HH:mm');
      if (isTimeInRange(currentTime, prefs.quiet_hours_start, prefs.quiet_hours_end)) {
        return false;
      }
    }

    return true;
  }
  ```
