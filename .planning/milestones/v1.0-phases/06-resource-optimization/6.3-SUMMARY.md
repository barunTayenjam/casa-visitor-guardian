# Plan 6.3 Summary: Stream-on-Demand

**Executed:** 2026-03-19
**Status:** Complete

## Changes Made

### Stream Lifecycle with Inactivity Timeout

Updated `stopStream` function in server/src/streams/rtspManager.ts:
- When all viewers disconnect, FFmpeg process stays running for 5 minutes
- If viewers reconnect within timeout, the inactivity timer is cancelled
- Only after timeout expires is FFmpeg killed

### Viewer Reconnection Handling

Updated `requestStream` handler:
- Checks for pending inactivity timeout when viewer reconnects
- Cancels the timeout if viewer is reconnecting
- Logs when timeout is cancelled

## Architecture

```
Viewer Disconnects → Start Inactivity Timer (5 min)
                              ↓
              New Viewer Connects? ──Yes──→ Cancel Timer
                              ↓
                            No
                              ↓
                    Timer Expires → Kill FFmpeg
```

## Configuration

```typescript
// Default values
const STREAM_INACTIVITY_TIMEOUT = 5 * 60 * 1000; // 5 minutes
```

## Resource Savings

| Scenario | Before | After |
|----------|--------|-------|
| Brief disconnect (< 5 min) | FFmpeg restart | No restart |
| Rapid connect/disconnect | Multiple restarts | Single timeout |
| Idle time | FFmpeg always running | Auto-stop after 5 min |

## Benefits

1. **Prevents FFmpeg thrashing**: Brief disconnections don't trigger expensive restarts
2. **Reduces idle resource usage**: FFmpeg stops after 5 minutes of no viewers
3. **Better user experience**: Quick reconnections don't wait for FFmpeg restart
4. **Lower CPU usage**: Reduced context switching from frequent restarts
