```
  - FFmpeg: 4 threads/camera, 35% CPU/camera
```

### 2. Bottleneck Analysis
**Identified 5 critical bottlenecks:**
1. **High CPU (71%)**: FFmpeg encoding consuming 4 threads per camera
2. **Memory bloat (897MB)**: Node.js + FFmpeg processes per camera
3. **No Docker limits**: Unlimited resource access causing system stress
4. **Continuous streaming**: No stream-on-demand optimization
5. **Continuous detection**: OpenCV running 28% CPU even when idle

### 3. Optimization Targets
**Established measurable targets:**
| Metric | Current | Target |
|--------|---------|--------|
| Total CPU | 103% | <50% (1 core) |
| Total Memory | 1.45 GB | <1.0 GB |
| Per-camera CPU | 35% | ~15% |
| Per-camera Memory | 120 MB | ~60 MB |

## Findings

### FFmpeg Configuration
**Current Settings:**
```typescript
"-threads", "4",     // 4 threads per camera
"-r", "4",           // 4 FPS
"-vf", "scale=1920:1080",
"-q:v", "5"          // Quality 2-31 scale
```

**Per-Camera Resource Usage:**
- CPU: ~35% (encoding overhead)
- Memory: ~80-120 MB
- 2 FFmpeg processes total (one per camera)

### Docker Configuration
**Current State:** No resource limits set
- Risk of swap thrashing under load
- Containers can consume unlimited system resources
- No isolation or priority controls

### OpenCV Detection
**Detection Pipeline Timing:**
- Motion detection: ~5ms per frame
- Object detection: ~50ms per detection
- Face detection: ~30ms per face
- Total: ~85ms per full detection cycle

**Continuous Operation:** 28% CPU even when idle

---

## Optimization Roadmap

**Prioritized recommendations:**
1. **FFmpeg**: Reduce threads (4→1), lower quality, reduce FPS (4→2)
2. **Docker**: Add memory/CPU limits to prevent resource exhaustion
3. **OpenCV**: Implement motion-triggered detection (not continuous)
4. **Streaming**: Add stream-on-demand with inactivity timeout
5. **Adaptive quality**: Scale resolution based on viewer count
